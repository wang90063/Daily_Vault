# 分享计划：我怎么使用 OrbitOS，以及怎么配置 Codex / Claude

> [!abstract]
> 这次分享面向第一次接触 [[OrbitOS]] 的人，重点不是把功能念一遍，而是讲清楚三件事：我为什么要做这套系统、我每天真实是怎么用的、`Codex` 和 `Claude` 是怎么被配置到可用状态的。

## 为什么我要做这个项目

- 我最早遇到的问题不是没有记录工具，而是记录之后没有后续：想法丢进笔记里，项目和研究分散在不同地方，过几天就找不到上下文。
- 另一个问题是每天启动成本很高。打开 Obsidian 以后，脑子里有很多事，但不知道应该先处理收件箱、推进项目，还是继续研究。
- 我想要的不只是一个笔记库，而是一套从捕捉到沉淀都能跑通的个人工作流，让 AI 真正参与管理，而不是只在旁边回答问题。
- 所以 [[OrbitOS]] 的目标不是“做一个更复杂的模板系统”，而是把我的日常工作拆成清晰的轨道，让信息始终处于流动状态。

## 这套系统解决了我什么问题

- `00_收件箱` 解决“先记下来再说”的问题。我不需要一开始就决定这是项目、研究还是临时想法，先放进去，后面再处理。
- `10_日记` 解决“今天从哪里开始”的问题。每日笔记是锚点，我会把当天要推进的事情、正在做的项目、关键进展都挂到这里。
- `20_项目` 解决“执行过程没有连续性”的问题。项目不是一堆分散任务，而是有背景、动作和进展的连续记录。
- `30_研究` 和 `40_知识库` 解决“学过的东西留不下来”的问题。研究笔记负责承载上下文，知识库负责提炼成可复用的概念。
- `90_计划` 解决“直接开做容易跑偏”的问题。复杂工作先让 AI 起草方案，我确认以后再执行，避免一上来就陷进细节。
- 这套流转本质上就是 [[OrbitOS流转模型]]：捕捉 -> 处理 -> 执行 -> 沉淀。分享时这一段要讲得尽量直白，不要抽象化。

## 我一天是怎么用 OrbitOS 的

- 一天开始前，如果脑子里突然冒出一个想法，我会先丢到 `00_收件箱`，不做分类，不追求完整。
- 正式开始工作时，我会先看今天的日记，或者先用 `/start-my-day` 把收件箱、活跃项目和今天的重点重新拉起来。
- 如果某条想法已经足够清晰，并且需要推进，我会用 `/kickoff` 把它变成正式项目；如果它更像一个需要调研的问题，我会走 `/research`。
- 在执行过程中，我不会把所有内容都塞进一个地方，而是让项目、研究、知识条目各自承担不同职责，再通过双链把它们连起来。
- 到了晚上或者一个阶段结束的时候，我会把当天的关键进展回写到日记或者项目里，这样第二天打开系统时能立刻接上昨天的上下文。
- 这部分分享最好按“真实的一天”来讲，而不是按命令列表来讲。重点是让别人看到这套系统如何减少切换成本。

## 我会现场演示什么

- 先展示一个真实的 `00_收件箱` 条目，说明我不会在捕捉阶段追求结构化。
- 再展示一篇 `10_日记`，解释为什么我把每日笔记当作锚点，以及它怎么把当天相关事项串起来。
- 然后展示一个 `20_项目` 或 `30_研究` 的例子，重点看它如何链接回 [[OrbitOS]] 的整体流转，而不是单独存在。
- 最后切到配置层，说明为什么同一套目录和规则能同时驱动 `Codex` 与 `Claude`。
- 演示时尽量走一个完整链路：收件箱输入 -> 日记承接 -> 项目或研究展开 -> 关键配置解释。

## Codex 和 Claude 我是怎么配置到可用的

- 第一层是环境：用 Obsidian 打开整个仓库，把 Vault 当作一个可直接操作的工作目录，而不是只看单篇笔记。
- 第二层是终端入口：我会在 Vault 根目录运行 `Codex CLI` 或 `Claude Code`，这样它们看到的是同一套目录结构和同一批系统文件。
- 第三层是行为约束：`AGENTS.md` 负责定义高层规则，比如目录职责、输出语言、项目与知识的组织方式。
- 第四层是技能体系：真正的技能唯一源目录在 `.agents/skills/`，像 `.codex/skills`、`.claude/skills` 这种路径只是兼容入口，不是重复维护的源文件。
- 第五层是工作流触发：我平时不是靠“长提示词”硬控模型，而是把常见任务做成技能，让不同 CLI 进入同样的 OrbitOS 约束。
- 这一段分享不需要展开底层原理，只要让听众知道三个关键点：装什么、从哪里启动、规则写在哪。

## 我现在的使用分工与习惯

- `Codex` 更适合在当前仓库里做结构化整理、文件操作和明确目标下的执行，因为它对本地文件和修改链路更直接。
- `Claude` 更适合长对话、开放式探索和需要连续追问的场景，尤其是我还没把问题想清楚的时候。
- 但两者不是两套系统，而是同一套 [[OrbitOS]] 规则下的两个入口。目录结构、技能来源、笔记组织方式都保持一致。
- 我的重点不是比较哪个模型更强，而是让它们都进入一个可复用、可追踪、可沉淀的工作流。

- 一篇真实的日记，能看出当天任务、项目链接和上下文承接。
- 一个真实的项目或研究笔记，能展示双链和内容沉淀。
- 一张仓库目录截图，方便快速解释 `00_收件箱`、`10_日记`、`20_项目`、`30_研究`、`40_知识库`、`90_计划`、`99_系统` 的分工。
- `AGENTS.md` 和 `.agents/skills/` 的截图或片段，说明规则和技能是怎么组织的。
- 我最常用的 2 到 3 个命令，比如 `/start-my-day`、`/kickoff`、`/research`。
