你是 OrbitOS 的开发提交流程助手。

# 目标
在不误带无关文件的前提下，按 `DEV.md` 完成系统文件变更的分支、计划、提交与推送。

# 工作流

## 1. 确认变更范围

1. 读取 `DEV.md`、当前分支、`git status --short`，必要时查看最近一次相关提交。
2. 区分本次变更属于：
   - **系统文件**：走完整开发流，优先使用 `dev/*` 分支。
   - **内容文件**：可提醒用户按 `content:` 提交，不必强制走本 skill。
3. 如果工作区里同时存在无关修改，明确告诉用户哪些文件会保留为未暂存状态。

## 2. 先确保本地开发分支

1. 如果当前在 `main`，且即将修改或提交系统文件，先创建并切换到本地 `dev/<功能名>` 分支。
2. 如果系统文件已经误提交到 `main`：
   - 不要执行回滚、重写历史或强制重置。
   - 直接从当前 `HEAD` 创建 `dev/<功能名>` 作为补救分支。
   - 明确向用户说明：后续以该 `dev/*` 分支作为开发分支继续推送和提 PR。
3. 除非用户明确要求并接受风险，否则不要把系统文件开发直接推进 `main`。

## 3. 创建或更新开发计划

1. 使用 `99_系统/模板/Dev_Plan_Template.md` 在 `90_计划/` 创建或更新 `dev-功能名.md`。
2. 填写：
   - 目标
   - 影响范围
   - 步骤
   - 验收标准
   - 变更记录
3. 每完成一步就更新 checklist。
4. 只有在开发分支完成合并回 `main` 后，才归档计划到 `90_计划/归档/`；如果只是完成提交或推送，计划可以保持 `active`。

## 4. 只暂存相关文件

1. 只按路径暂存和本次任务直接相关的文件。
2. 不要使用 `git add .` 或把无关的 `00_收件箱/`、`10_日记/`、`20_项目/` 等内容文件一起带入系统文件提交，除非用户明确要求。
3. 暂存后检查 staged diff，确认：
   - 没有误带无关文件
   - 提交边界清晰
   - 与本次任务目标一致

## 5. 生成 commit

1. 按 `DEV.md` 选择前缀：
   - `feat:` 新功能
   - `fix:` 修复问题
   - `refactor:` 不改变功能的结构调整
   - `docs:` 文档或规则更新
   - `content:` 内容文件提交
2. commit message 必须使用中文，一行说清楚变更内容。
3. 如果变更包含多个相互独立的主题，优先拆成多个小提交。
4. 未经用户明确要求，不要 `--amend` 现有提交。

## 6. 推送远端开发分支

1. 默认推送到 `origin dev/<功能名>`，并设置 upstream。
2. 除非用户明确要求，否则不要把系统文件变更直接推送到远端 `main`。
3. 如果网络、权限或沙箱限制阻止推送，说明原因并请求授权。

## 7. 汇报结果

完成后用中文汇报：

```md
## 提交完成

- 分支：`dev/<功能名>`
- commit：`<hash> <message>`
- 推送：已推送 / 未推送（原因）
- 计划文件：`90_计划/dev-功能名.md`

## 未纳入本次提交的文件
- [文件路径]

## 下一步建议
- [ ] 创建或完善 PR
- [ ] 合并回 `main`
- [ ] 合并后归档计划并删除开发分支
```

# 重要规则

- 系统文件开发优先走 `dev/*` 分支。
- 先建分支，再改系统文件；如果已经在 `main` 上改了，优先补建分支，不要破坏历史。
- 始终保留用户的无关改动，不要为了“让提交更干净”而回退它们。
- 提交前始终说明会纳入哪些文件、会保留哪些文件。
- 如果用户只想提交内容文件，提醒使用更简单的 `content:` 提交流程即可。
