# 双代理模式

## 定义

双代理模式是一种工作流架构模式，将复杂任务分解为两个独立的代理阶段：**规划代理（Planning Agent）** 和 **执行代理（Execution Agent）**。两个代理通过一个中间计划文件进行通信，执行代理在全新的上下文中启动，仅读取计划文件作为输入。

## 要点

- **上下文隔离**: 执行代理不继承规划代理的上下文，保持清洁的工作环境
- **用户检查点**: 在规划和执行之间插入用户审阅环节，允许修改计划
- **职责分离**: 规划代理专注于策略和结构设计，执行代理专注于具体实现
- **计划文件作为契约**: 中间计划文件是两个代理之间的唯一通信接口
- **Token 效率**: 避免在执行阶段携带大量不必要的上下文，减少 Token 消耗

## 示例

在 OrbitOS 中，`kickoff` 和 `research` 技能都采用双代理模式：

**kickoff 工作流**:
```
用户 → Orchestrator → Planning Agent → 创建计划文件
                                      ↓
                                用户审阅/修改
                                      ↓
     Orchestrator → Execution Agent → 读取计划 → 创建项目笔记
```

**规划代理任务**:
- 搜索相关现有项目和笔记
- 识别相关领域
- 设计项目结构
- 创建计划文件 `90_计划/Plan_YYYY-MM-DD_Kickoff_<ProjectName>.md`

**执行代理任务**:
- 仅读取计划文件（全新上下文）
- 根据计划创建项目笔记
- 链接到日记
- 归档计划和收件箱文件

**优势体现**:
1. 规划阶段可以广泛搜索和分析，不担心上下文膨胀
2. 用户可以修改计划中的任何细节（时间线、优先级、结构等）
3. 执行阶段专注于文件创建，不被规划阶段的大量信息干扰
4. 如果执行失败，可以重新运行执行代理而无需重新规划
