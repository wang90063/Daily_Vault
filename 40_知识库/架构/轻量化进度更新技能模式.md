# 轻量化进度更新技能模式

## 定义

轻量化进度更新技能模式，是指为“用户只想同步当前进展”这一类窄任务设计的 Skill 工作流模式。它的核心目标不是推进任务，而是用最小动作把项目现状、已完成内容、当前重点和阻塞整理清楚，并在用户明确要求写回时，把进展与待办、步骤和中间物建立结构化关联。

## 要点

- **边界优先**: 先区分“状态同步”和“任务执行”，默认按更保守的状态同步处理
- **最小输出**: 默认只提炼当前状态、已完成、当前重点和阻塞/待确认，不自动扩展成计划、方案或执行
- **写回有结构**: 用户要求落库时，不能只追加孤立日志，要把进展挂到对应待办和对应步骤
- **中间物可追溯**: 截图、草图、临时图等中间物应优先归档，再在 `Progress` 中建立链接
- **不确定时直接问**: 无法高置信度判断该连哪条待办或哪个步骤时，直接向用户确认，而不是替用户拍板

## 背景与问题来源

这个模式来自一次典型失误：用户原本只是想“更新一下项目进度”，但 agent 把输入误判成了需要继续展开、规划甚至推进执行的任务。该案例暴露出一个常见问题：

`进度同步` 不等于 `请求执行`。

因此，Skill 的设计重点从“如何更主动”转向“如何更贴边界”，把默认行为收敛为轻量整理，而不是主动扩展。

## 设计演化

### 第一阶段：定义最小边界

最初的 `/update-progress` 聚焦两件事：

1. 把用户输入识别为状态同步而不是任务推进
2. 只输出简短、可直接记录的进展内容

这一阶段确定了最小必要要素：

- 当前状态
- 已完成
- 当前重点
- 阻塞/待确认

### 第二阶段：补齐写回规则

在实际使用中，仅仅“写一条进展”还不够。用户进一步提出三个结构化要求：

1. 中间物应归档，而不是继续放在收件箱里
2. 日记里的 `progress` 应该与某条待办事项对应
3. 项目里的 `Progress` 应该与 `Actions` 中的某个步骤对应

于是，`/update-progress` 新增了写回规则：

- 中间物默认可从 `00_收件箱/` 归档到 `99_系统/归档/收件箱/YYYY/MM/`
- 日记写回时，优先关联到明确的待办事项
- 项目写回时，优先关联到明确的阶段或步骤
- 如果无法高置信度判断，就直接向用户发问

### 第三阶段：形成可复用交互

在规则稳定后，这个模式的推荐交互被收敛为：

1. 先压缩进展内容
2. 如果涉及写回，先判断是否有中间物需要归档
3. 再判断进展对应哪条待办、哪个步骤
4. 不确定时，一次性问清楚

推荐问题句式：

`这条进展要连到今日日记的哪条待办，以及项目里的哪个步骤？`

## 标准 Pipeline

```text
用户更新进展
  -> 判断：状态同步 / 任务执行 / 混合型
  -> 提取：当前状态 / 已完成 / 当前重点 / 阻塞
  -> 输出轻量进展
  -> 若用户要求写回：
       -> 归档中间物
       -> 关联今日日记待办
       -> 关联项目步骤
       -> 写入日记与项目 Progress
```

## 关键规则

1. 遇到“更新一下进度”“先记一下”“不用做别的”时，默认进入最小动作模式。
2. 没有明确授权时，不把进度更新升级成计划、执行、落库或额外文档。
3. 写回日记时，`progress` 要与待办事项建立对应关系。
4. 写回项目时，`Progress` 要与 `Actions` 中的阶段或步骤建立对应关系。
5. 中间物优先归档，再在进展文字中链接归档结果。
6. 如果关联关系不明确，直接问用户，不做隐式猜测。

## 示例

以 [[分层分级技术图]] 的一次更新为例：

- 中间物：`framework.png`
- 归档位置：`99_系统/归档/收件箱/2026/03/framework.png`
- 日记关联：今日日记中“完成分层分级技术图第二轮整理”这条待办
- 项目关联：`阶段1: 输入输出梳理`

最终形成的进展表达不是孤立记录，而是：

- 在日记里明确这条日志对应哪条待办
- 在项目里明确这条 `Progress` 对应哪个步骤
- 在进展文字中链接归档后的中间物

## 价值

- 降低 agent 把小请求升级为大任务的概率
- 让项目进展具备“来源可追溯、位置可对应、结构可延展”的特征
- 让日记、项目、归档三者之间形成可导航的闭环
- 为后续的周报整理、项目复盘和资料回溯提供清晰基础
