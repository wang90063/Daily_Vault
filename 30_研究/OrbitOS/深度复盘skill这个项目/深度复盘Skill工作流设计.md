# 深度复盘Skill工作流设计

## 概述
这份设计文档对应 [[深度复盘Skill]] 项目计划中的“阶段2：工作流设计”。目标不是直接实现 skill，而是先把输入项、处理流程、输出格式和模式切换规则定义清楚，确保后续实现阶段有明确边界。

结合最新研究结论，本项目的主模型调整为 `GRAI + KISS`：
- `GRAI` 负责周复盘、项目复盘、重大事件复盘等深度场景
- `KISS` 负责日复盘和轻量周回顾等行动场景

## 设计目标
- 让用户能根据周期和事件复杂度，在一次调用中进入合适的复盘深度
- 保证复盘结果既能形成洞察，也能收敛成少量下一步行动
- 让输出可以自然回写到日记、项目和知识库，而不是停留在一次性对话
- 避免复盘变成自责问卷或任务爆炸器

## 输入项设计
### 默认扫描范围
为了避免每次复盘都扫描整个仓库，Skill 必须遵守“最小必要扫描”原则。

默认规则：
- 如果用户明确给出文件、项目或时间范围，只扫描这些明确指定的对象
- 如果用户没有给出文件路径，先从与场景最相关的单一目录开始，而不是全库扫描

推荐默认扫描目录：
- **日复盘**：默认只扫描 `10_日记/` 中当天日记和`00_日记/` 中记录自己的内容
- **周复盘**：默认扫描 `10_日记/` 中本周涉及的日记文件
- **项目复盘**：默认扫描目标项目在 `20_项目/` 下的项目笔记；如果用户明确点名某个研究笔记，再补扫对应研究文件
- **重大事件复盘**：默认扫描用户提供的事件材料，或与该事件直接相关的单个项目/日记文件

禁止默认行为：
- 不默认扫描整个 `20_项目/`
- 不默认扫描整个 `30_研究/`
- 不默认扫描整个 Vault

### 扫描升级规则
只有在当前上下文不足以支持复盘时，才逐层扩大扫描范围：

1. 先读用户明确提供的文件或对象
2. 再读当前周期最相关的日记或项目笔记
3. 如仍不足，再补充直接关联的研究笔记或知识库概念
4. 如果还不够，不继续盲扫，而是要求用户指定范围

### 必填输入
- **时间范围**：今天 / 本周 / 某个项目阶段 / 某次具体事件
- **复盘对象**：个人整体、单个项目、单次会议、一次失败或成功事件
- **事实材料**：发生了什么，尽量按事件或结果写

### 条件必填输入
- 当进入 `GRAI` 时，补充：
  - **目标定义**：原目标是什么，是否符合 SMART，成功标准是什么
  - **结果对比**：实际结果、差距类型、证据材料
- 当进入 `KISS` 时，补充：
  - **当前动作清单**：Keep / Improve / Stop / Start 的初始草稿
  - **当期关注点**：本轮最想调整的 1 个重点

### 辅助输入
- **感受与精力**：高点、低点、波动时段、触发因素
- **上下文链接**：相关日记、项目、会议记录、任务清单、截图
- **特殊约束**：时间限制、优先级变化、外部条件变化

## 处理流程设计
### 步骤0：确定扫描范围
在进入模型前，先确定本轮复盘读取哪些文件。

执行顺序：
- 先读用户明确指定的文件、项目、时间范围
- 若未指定，再按“默认扫描范围”选择最小必要目录
- 若信息仍不够，再按“扫描升级规则”逐层补充

这一步的目标是控制上下文，避免“为了保险把整个项目扫一遍”。

### 步骤1：判断模式
先根据周期和问题类型判断进入 `GRAI` 还是 `KISS`。

进入 `GRAI` 的条件：
- 时间范围大于一周
- 涉及项目阶段性结果、重大偏差或关键事件
- 用户明确想找根因、提炼洞察或调整方向
- 连续多轮 `KISS` 没有稳定的 Keep，说明问题已不是局部行动层

进入 `KISS` 的条件：
- 日常收尾、每日复盘、轻量周回顾
- 用户主要想判断哪些做法继续、哪些微调、哪些停止、哪些尝试
- 当前问题更像行动节奏问题，而不是方向性问题

### 步骤2：整理事实与证据
无论进入哪个模型，都先分离“事实”和“解释”：
- 先列事件、结果、证据
- 再写感受和精力
- 最后再进入分析或行动判断

这一步的目的是防止复盘一开始就进入人格评价或模糊归因。

### 步骤3A：执行 GRAI
当模式判定为 `GRAI` 时，按下面链路处理：

1. **Goal**
   - 回看当时目标是什么
   - 检查目标是否具体、可衡量、现实
2. **Result**
   - 对比实际结果与预期结果
   - 判断是正差距还是负差距
3. **Analysis**
   - 列举所有可能原因，不提前批判
   - 对关键原因持续追问“五个为什么”
   - 同时扫描工作与生活两个维度，避免单维误判
4. **Insight**
   - 提炼能复用的模式判断
   - 明确哪些认知、策略或定位需要调整

### 步骤3B：执行 KISS
当模式判定为 `KISS` 时，按下面链路处理：

1. **Keep**
   - 先找继续放大的正确做法
   - 即使整体结果一般，也优先确认哪些环节是对的
2. **Improve**
   - 明确方向是对的，但哪些细节需要调优
3. **Stop**
   - 停掉低价值、无效或当前优先级不成立的动作
4. **Start**
   - 只在 Stop 和优先级清楚后，再启动新的尝试
   - Start 必须绑定假设、触发场景和检查点

### 步骤4：行动收敛
无论使用哪一个模型，最后都要进入统一的行动收敛层：
- 最多保留 1 到 3 条行动实验
- 每条行动必须写清触发条件或检查时间
- 如果行动过多，优先删减而不是全部保留

### 步骤5：回写上下文
输出完成后，按用户授权程度决定是否回写：
- 写入今日日记
- 关联现有项目的 Progress
- 若形成可复用模式，再沉淀到知识库

## 输出格式设计
### 标准输出
- **复盘概览**：一句话总结本轮复盘的主结论
- **事实与结果**：本周期关键事件、关键结果、差距
- **核心洞察或行动判断**：
  - `GRAI` 输出 1 到 2 条洞察
  - `KISS` 输出 Keep / Improve / Stop / Start
- **下一步实验**：最多 3 条
- **关联链接**：相关日记、项目、知识库概念

### GRAI 输出偏重
- 目标是否成立
- 差距来自哪里
- 哪个洞察值得复用到下一轮

### KISS 输出偏重
- 哪个做法继续放大
- 哪个细节优化
- 哪个动作停止
- 哪个新尝试开始

## 模式衔接规则
`GRAI` 和 `KISS` 不是互斥关系，而是上下层协作：
- `GRAI` 负责找到长期模式和方向调整点
- `KISS` 负责把洞察转成日常行动
- 若连续多轮 `KISS` 仍没有稳定 Keep，则升级到 `GRAI`
- 若 `GRAI` 得到洞察后没有进入行动实验，则下一轮默认切回 `KISS`

## 边界与防错
- 不把日复盘强行升级成深度根因分析
- 不把深度复盘写成问题清单而没有洞察
- 不允许一次复盘生成大量任务
- 分析阶段禁止直接归责，优先使用 [[无责复盘原则]]
- 涉及个人状态时，必须扫描 [[个人复盘双维度模型]]，避免只看工作维度

## 阶段结论
按照项目计划，阶段2的三个核心问题已经有了明确答案：

1. **输入项**：时间范围、复盘对象、事实材料是主输入，目标/结果或 KISS 草稿按模式补充
2. **处理逻辑**：先判定 `GRAI` 或 `KISS`，再统一进入行动收敛与上下文回写
   - 在此之前，必须先确定默认扫描范围，避免全库扫描
3. **输出格式**：统一输出“概览 + 事实结果 + 洞察/行动 + 下一步实验 + 关联链接”

这意味着项目可以在阶段2结束后，再进入阶段3的 skill prompt 与工作流实现。
