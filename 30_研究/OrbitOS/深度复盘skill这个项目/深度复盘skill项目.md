# 深度复盘skill这个项目

## 概述
本研究面向 [[深度复盘Skill]] 项目，目标不是泛泛介绍“复盘”，而是为 OrbitOS 里的一个可执行 Skill 选出合适的方法论组合、输入结构、分析流程和输出格式。

结合你在计划文件中的回答（知识水平偏初级、希望直接应用到自己身上、理论与示例同样重要），一个适合个人使用的“深度复盘”不应只照搬团队 Sprint 回顾，而应同时满足：
- 足够轻，能每天或每周稳定执行
- 足够深，能从事件事实进入情绪、判断和行动
- 可落库，能把结论回写到日记、项目和知识库
- 不走偏成自责日记，而是生成下一步实验

## 核心概念
- [[复盘问题梯度]]：用“事实 -> 感受 -> 解释 -> 决定”控制提问深度
- [[复盘行动闭环]]：把洞察转成少量、可验证、可回看的行动
- [[无责复盘原则]]：把注意力从自我指责转向模式识别与改进
- [[个人复盘双维度模型]]：同时审视工作与生活，避免单一维度失真
- [[双代理模式]]：可借鉴其“计划 -> 审阅 -> 执行”的清洁上下文设计
- [[轻量化进度更新技能模式]]：提醒 Skill 必须守边界，不把“记录一下”强行升级成“深度分析”

## 工作原理
### 1. 研究结论：建议采用“ORID 主干 + KPT 输出 + AAR 校验”的混合模型
- ORID 负责提问顺序：先事实，再感受，再解释，最后行动
- KPT 负责收敛输出：Keep / Problem / Try 易读、易复用，也适合长期积累
- AAR 负责避免空泛：用“预期 vs 实际”“为什么有差异”“下次怎么做”校准行动质量
- 这个组合是推断：它不是某个单一来源直接给出的现成模板，而是基于多种复盘框架互补性做出的项目级设计选择

### 2. 输入层
建议 Skill 的输入至少覆盖 6 类信息：
1. 今日或本周发生了什么
2. 哪些结果符合预期，哪些不符合
3. 工作维度的亮点、阻塞、决策与协作问题
4. 生活维度的精力、情绪、节奏与恢复情况
5. 证据材料：日记、项目进展、截图、链接、任务清单
6. 想继续保持、想停止、想尝试的事项

推荐输入结构：
- 时间范围：今天 / 本周 / 某个事件
- 复盘对象：个人整体 / 单个项目 / 某次关键事件
- 事实清单：发生了什么，尽量按事件写
- 情绪与精力：高点、低点、触发因素
- 差异对比：原计划 vs 实际结果
- 想法草稿：先不要求结论，只先收集线索

### 3. 分析层
建议 Skill 的内部分析分四步：
1. 事实归档：区分“发生了什么”和“我怎么理解”
2. 双维度扫描：按工作 / 生活分别看高点、低点、消耗与恢复
3. 模式提炼：识别重复出现的触发条件、阻塞点、有效做法
4. 行动收敛：只留下 1-3 个下一个实验，不生成过多任务，控制行动总量

一个可执行的提问链可以是：
- Objective：今天我具体做了什么？有哪些事件是确定发生的？
- Reflective：哪些时刻让我有能量，哪些时刻让我明显下沉？
- Interpretive：背后的模式是什么？哪些决策、环境或习惯在重复影响结果？
- Decisional：接下来我准备保留什么、修正什么、实验什么？

### 4. 输出层
建议 Skill 的标准输出由五部分构成：
1. 复盘概览：一句话总结本次周期
2. 工作维度：进展、卡点、协作 / 决策反思
3. 生活维度：精力、作息、情绪、恢复
4. 洞察与模式：真正值得记住的 2-4 条观察
5. 下一步实验：最多 3 条，每条带触发场景或时间边界

如果需要更贴 OrbitOS，可以在正文后附加：
- 关联项目：[[项目名]]
- 关联日记：[[2026-03-18]]
- 关联知识：若发现可复用模式，再抽出知识库概念

### 5. Skill 形态建议
对于 [[深度复盘Skill]]，更适合做成“双层模式”，而不是所有情况下都强制深度：
- 轻量模式：3-5 分钟，适合每日收尾，只输出 KPT 或“三问”
- 深度模式：15-30 分钟，使用 ORID 梯度，适合周复盘、事件复盘或情绪波动较大的阶段
- 写回模式：在用户明确要求时，把结论同步到日记、项目 Progress 或知识库

这种分层设计可以降低用户负担，也能避免每次调用都变成沉重长问卷。

## 实现蓝图（续推）
### 1. 触发词与场景守门
建议把 `深度复盘 skill` 的典型触发词分成三组，并在入口先做一次“是否需要深度”的守门判断：

- **深度触发词**：`深度复盘`、`事件复盘`、`帮我复盘这次失败/偏差`、`找根因`
- **轻量触发词**：`日复盘`、`收尾复盘`、`今天哪些该继续`
- **写回触发词**：`写进今日日记`、`同步项目进展`、`沉淀成知识点`

入口守门规则：
- 若用户表述接近“只更新状态”，应提示使用 [[轻量化进度更新技能模式]]
- 若用户目标是“复盘 agent 失误”，应切到 `reflect`，避免重复做错误归因
- 若用户只是知识问答，应切到 `ask`，避免把问答升级为复盘流程

### 2. 与现有 skills 的边界差异
| 对比对象 | 主要目标 | 默认动作 | 不该做什么 |
|---|---|---|---|
| 深度复盘 skill | 提炼模式与下一步实验 | 事实-分析-行动收敛 | 直接变成项目计划拆解 |
| `reflect` | 复盘一次 agent 交互失误 | 证据归因 + 防错规则 | 延展到日常生活/项目全复盘 |
| `update-progress` | 轻量同步状态 | 当前状态 + 已完成 + 阻塞 | 未授权时自动扩展执行 |
| `ask` | 快速答疑 | 直接回答问题 | 建立重型工作流与落库 |

边界判断优先级建议：
1. 先判断用户是否只要“状态同步”
2. 再判断是否是“agent 失误复盘”
3. 再进入“个人深度复盘”

### 3. 默认扫描范围与升级规则
推荐使用最小必要扫描矩阵（详见 [[复盘Skill默认扫描矩阵]]）：

- **日复盘（KISS）**：默认只读 `10_日记/` 当天文件 + 用户指定材料
- **周复盘（KISS->GRAI）**：默认读本周日记 + 目标项目单文件
- **事件复盘（GRAI）**：默认读事件直接证据 + 一个关联项目文件

升级规则：
1. 先读用户明确指定对象
2. 信息不足时，只补一层直接关联材料
3. 仍不足时停下确认，不盲扫整个 Vault

### 4. 输出结构与回写规则
建议统一输出骨架，并把“回写”做成显式可选项（详见 [[复盘Skill输出回写策略]]）：

1. 复盘概览（一句话）
2. 事实与结果（证据化）
3. 模式判断（GRAI 洞察或 KISS 四象限）
4. 下一步实验（最多 3 条，含触发条件/检查点）
5. 关联链接（项目、日记、知识）

回写优先级：
- 用户明确授权时才写回
- 先写 `10_日记/YYYY-MM-DD.md`
- 与项目强相关时再写 `20_项目/...` 的 `Progress`
- 只有可复用模式才新增 `40_知识库/...`

### 5. 落地路径（MVP）
- **MVP-1：入口守门**：先实现触发词分类与边界分流
- **MVP-2：双模式处理**：实现 `KISS` 与 `GRAI` 两条处理链
- **MVP-3：输出与回写**：实现统一输出骨架 + 授权式回写
- **MVP-4：案例校准**：用示例回放验证结果是否“可执行而不臃肿”

这里给出一个最小示例：

```markdown
时间范围：2026-03-18
复盘对象：个人工作日
事实：
- 完成了方法论调研
- 没有推进项目结构探索 Task 3.1
- 下午精力明显下降

感受：
- 上午有掌控感
- 下午出现拖延和分心

解释：
- 上午任务清晰、反馈快
- 下午任务切换成本高，且没有先定义最小下一步

行动：
- Keep：上午优先处理结构清晰的任务
- Problem：下午进入模糊任务时容易失速
- Try：每天 14:00 前先为下午任务写出“最小下一步”
```

## 最佳实践
- 先收事实，再下判断；不要把解释当事实写进去
- 每次复盘最多收敛 1-3 个行动实验，避免任务爆炸
- 工作与生活分开看，再在最后合并模式，避免全混在一起
- 优先链接证据：日记、项目进展、截图、任务，而不是只写感想
- 输出中明确“继续保持什么”，不要只盯着问题
- 如果复盘涉及挫败或错误，优先用 [[无责复盘原则]] 描述情境和条件，而不是评价人格

## 常见陷阱
- 直接问“今天哪里做得不好”，会过早把用户推向自责
- 问题过多且没有层次，导致用户写到一半就放弃
- 洞察很多，但没有转成少量可执行实验
- 只复盘工作，不复盘精力与生活节奏，容易误判根因
- 把项目进展同步和深度复盘混在一起，导致输出既不像复盘也不像状态更新
- 每次都要求完整深度模式，长期使用成本过高
