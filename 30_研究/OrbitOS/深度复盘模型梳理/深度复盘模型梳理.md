# 深度复盘模型梳理

## 概述
本次研究以 B 站字幕中的 GRAI/KISS 辅以现有知识库为素材，重新定义 OrbitOS 的深度复盘 Skill，明确 GRAI 负责重大/项目级复盘，KISS 负责日常行动复盘，把这两个模型摆在主干位置，同时将 ORID/KPT/AAR 保留为背景参照。

## 核心概念
- **GRAI**：Goal → Result → Analysis → Insight 的链路，强调先看目标再比结果、避免批判性归责、借助五个为什么追问直到形成可复用洞察。参照 [[复盘问题梯度]]。
- **KISS**：Keep / Improve / Stop / Start，引导行动复盘，把重点放在结果和计划，Keep 被视为最重要的词。参照 [[复盘行动闭环]]。
- **复盘问题梯度**：从事实到感受再到解释、决定的提问顺序，为 GRAI 的链路提供结构支持。
- **复盘行动闭环**：洞察必须转为 1-3 个可验证实验，为 KISS 提供输出规范。
- **无责复盘原则** 与 **个人复盘双维度模型**：确保复盘不陷入自责，工作与生活双维度并列。

## GRAI 模型详解
### Goal（目标）
GRAI 要求先回到当时设定的目标：为什么把目标定成这样、是否符合 SMART、是否清晰易量化。这个步骤决定后续的差距判断，符合 B 站中“先看目标再看结果”的叙述。
### Result（结果）
在目标基础上回顾实际结果，不仅看达成情况，还要分辨正差距还是负差距，为后续的分析提供对比。无需立即批判；正差距意味着找出做对了什么，负差距则继续挖掘根因。
### Analysis（分析）
按视频强调“列举所有可能原因、别急着批判”，再用五个为什么逐层拆解，借助 [[复盘问题梯度]] 的解释层去揭示模式和条件，确保包括行为、流程、环境等多个维度。
### Insight（洞察）
洞察由连续追问得出，通常是反常识、有独特判断力、且可在未来复用的结论。它扮演 GRAI 的知识产出，为下一轮复盘或 Skill 改进提供方向。
GRAI 适合季度、项目收尾或重大事件，需要安静的环境和充分的数据量，避免在日复盘中使用。

## KISS 模型详解
### Keep（保留）
KISS 的四个词中最重要的是 Keep，先确认做得对的事情，坚决放大成功实践，把 Good 路径继续带下去，避免总盯着问题。这个步骤与 [[复盘行动闭环]] 的“先写触发条件再确认执行”高度契合。
### Improve（改进）
在方向没错的基础上修正细节，比如产出率、协作流程、节奏节省等，属于趋势级判断，而非完全否定。
### Stop（停止）
识别低价值、消耗时间的行动，甚至在重新排优后明确“今天不要开始新任务”。Stop 与 Start 要相互制约，先停再开，避免任务线过长。
### Start（启动）
为下一轮尝试设定假设、触发点和预期指标，确保下一次复盘有清晰的检查点。Start 的输出也会成为未来 GRAI 深度复盘的输入。
KISS 此外不要求深入原因分析，重在行动节奏，适合每日/周的轻量复盘。

## 两者如何分层协作构建 Skill
1. **触发条件**：若用户选择“重大事件”“本周/本月回顾”或感觉结果产生较大差异，Skill 进入 GRAI；若是日常收尾、短周期尝试，则默认 KISS。
2. **数据流**：KISS 的 Keep/Improve/Stop/Start 输出，特别是 Start 中假设和 Stop 中需要暂停的事项，定期汇总后在 GRAI 的 Analysis 和 Insight 中被复盘，形成对长期模式的观察。
3. **流程桥接**：GRAI 的 Insight 中可以明示需要转入 KISS 的行动实验，KISS 的结果则反馈到下一次 GRAI 的 Result 层面，形成“洞察 → 行动 → 反馈 → 洞察”的闭环。
4. **节奏搭配**：GRAI 每月或关键事件一次，KISS 每日/周一次，Skill 需要提供选项让用户在开始时确定周期，也可由 AI 依据时间范围自动推荐。

## Skill 的输入 / 处理 / 输出草案
### 输入
- 时间范围：日、周、项目纪要、重大事件
- 复盘对象：个人工作/生活、某项目、某行动
- 目标设定：达成目标、其 SMART 特性、当时的 KPI
- 结果数据：实际产出、偏差、量化指标
- 感受与精力：什么时候有能量、什么时候低谷
- 证据清单：截图、任务卡片、会议记录、日志
- 今日/本周期的 Keep/Improve/Stop/Start（用于 KISS）

### 处理
- **步骤 1：判断复盘节奏**，若范围 > 一周或涉及关键结果，使用 GRAI；否则默认 KISS。
- **步骤 2：收集事实**，保持“事实 → 感受 → 解释 → 决定”的梯度，避免即时自责，必要时引用 [[无责复盘原则]]。
- **步骤 3：模型执行**，GRAI 逐步走 Goal → Result → Analysis → Insight，Analysis 中允许列举所有原因并用五个为什么拆解；KISS 按 Keep/Improve/Stop/Start 答复，强调先确认 Keep，再决定是否进入 Start。
- **步骤 4：行动设计**，只保留 1-3 条具体实验，带触发条件、指标、检查点，参考 [[复盘行动闭环]]。
- **步骤 5：链接上下文**，将洞察或行动关联到项目/日记/知识库，同时为下一次 Skill 调用预设情境。

### 输出
- **复盘概览**：一句话总结本周期的核心学到或感受。
- **核心洞察**：1-2 条来自 GRAI Insight 的模式发现，可供复用。
- **行动实验**：最多 3 条 Keep/Improve/Stop/Start 或由 Insight 引申的下一步实验，含触发点和检查时间。
- **关联上下文**：将输出回写到 [[深度复盘Skill]] 所关联的项目、日记或知识库概念。

## 与现有 ORID/KPT/AAR 方案的取舍建议
- **ORID vs GRAI**：两者结构相近，ORID 的 Objective/Reflective/Interpretive/Decisional 仍保留在 GRAI 的 Goal/Result/Analysis/Insight 中，但 GRAI 准确定义了结果差距与洞察复用，且更明确强调目标的 SMART 校验与持续追问。
- **KPT vs KISS**：KPT 的 Keep/Problem/Try 可看作 KISS 的兄弟，KISS 通过 Keep/Improve/Stop/Start 强调“先放大正确、再微调、再停，再试”，Stop 与 Start 的互为制约更适合轻量行动复盘。
- **AAR**：AAR 的“预期/实际/差异/下次”与 GRAI/KISS 的输出在本质上重叠，因此可作为 Skill 生成输出时的校验步骤，而不是独立主干。
- 当前 Skill 设计应以 GRAI + KISS 为主线，ORID/KPT/AAR 继续作为背景的提问梯度与输出校准工具，避免让用户在一次调用中面对过多模型混合。

## 最佳实践
- 在启动 GRAI 前先明确目标，必要时回去补写 SMART 细节。
- 将 KISS 的 Keep 条目列为最重要的输出，确保成功因素不会被问题吞没。
- Analysis 阶段要列举所有可能原因，再用五个为什么逐步拆解，避免立即归责。
- 连续使用 KISS 的 Stop/Start 形成小实验，并在下次 KISS 中检查执行情况。
- 每次复盘只带 1-3 个可验证行动，避免任务膨胀。
- 输出后立即关联一次现有项目/日记/知识库链接，确保产出落地。

## 常见陷阱
- 忽略目标就直接跳进结果描述，削弱 GRAI 洞察的深度。
- KISS 复盘只写 Problem 或 Stop，没列出 Keep，也没展望 Start，导致行动缺乏正向驱动。
- 在 Analysis 早期就批判责任人，违背 [[无责复盘原则]]。
- GRAI 用在日常收尾，把用户拖入过重的流程；应限定为大周期/关键事件。
- 复盘输出不包含验证时间或清晰触发点，导致行动无从检查。
