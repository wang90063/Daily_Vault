# OrbitOS Skills 工作流分析

## 概述

OrbitOS 的 Skills 系统是一套精心设计的工作流自动化机制，通过 `.agents/skills/` 目录下的 SKILL.md 文件定义各种专业化的工作流。每个 Skill 本质上是一个**智能代理的行为指令集**，定义了如何响应用户命令、如何操作文件系统、以及如何与其他 Skills 协同工作。

核心特点：
- **按需激活**: 通过 `/skill-name` 命令触发，避免上下文污染
- **专业化分工**: 每个 Skill 专注于特定场景（规划、研究、头脑风暴等）
- **双代理模式**: 复杂工作流采用规划+执行的两阶段架构
- **文件驱动**: 通过标准化的文件结构和 frontmatter 实现状态管理
- **工具链整合**: 灵活使用 Read、Write、Edit、Bash、WebSearch 等工具

## 核心技能分析

### 1. start-my-day - 每日规划工作流

**触发方式**: `/start-my-day`

**目标**: 帮助用户开始新的一天，通过回顾昨天进展、创建今日笔记、连接活跃项目来制定当日优先级。

**工作流步骤**:

1. **收集上下文（静默执行）**
   - 获取当前日期（YYYY-MM-DD 格式）
   - 读取昨天的日记 `10_日记/[yesterday].md`（如存在）
   - 提取未完成任务（未勾选的 `- [ ]` 项）
   - 搜索 `20_项目/` 中 `status: active` 的项目
   - 检查 `00_收件箱/` 中 `status: pending` 的条目
   - 分析并优先级排序（时间敏感项、3天未更新的停滞项目）

2. **用户交互（使用 AskUserQuestion 工具）**
   - 问题1: "今天的主要目标是什么？"（基于活跃项目提供选项）
   - 问题2: "有什么新想法或任务吗？"（自由文本输入）
   - 问题3: "有什么阻碍或顾虑吗？"（自由文本输入）

3. **创建今日笔记**
   - 检查 `10_日记/YYYY-MM-DD.md` 是否存在
   - 如存在：读取并更新（保留现有内容）
   - 如不存在：从模板 `99_系统/模板/Daily_Note.md` 创建
   - 填充内容：
     - **待办事项**: 昨天未完成任务 + 用户焦点 + 项目下一步行动
     - **日志**: 留空供用户填写
     - **备注**: 添加建议（时间敏感项、停滞项目、收件箱数量）
     - **相关项目**: 列出活跃项目及当前状态

4. **处理新想法（来自问题2）**
   - 检查想法是否已存在于项目或收件箱
   - 如为新想法，创建 `00_收件箱/[简短标题].md`
   - Frontmatter: `created`, `status: pending`, `source: start-my-day`

5. **呈现总结**
   - 输出中文总结：今日笔记链接、待办事项、活跃项目、新想法、收件箱状态
   - 提供快捷操作建议（`/kickoff`, `/research`）

**使用的工具**:
- **Read**: 读取昨天日记、项目文件、收件箱文件、模板
- **Write/Edit**: 创建或更新今日笔记、创建收件箱条目
- **Bash**: 搜索文件（`grep`, `find`）、检查文件存在性
- **AskUserQuestion**: 收集用户输入

**文件操作模式**:
- **读取**: `10_日记/[yesterday].md`, `20_项目/**/*.md`, `00_收件箱/*.md`, `99_系统/模板/Daily_Note.md`
- **创建/修改**: `10_日记/YYYY-MM-DD.md`, `00_收件箱/[NewIdea].md`
- **状态检查**: 通过 frontmatter 的 `status` 字段过滤

**关键设计原则**:
- 始终读取昨天的笔记（不假设为空）
- 优先级明确（时间敏感项优先）
- 标记停滞项目（3天未更新）
- 延续未完成任务
- 不覆盖现有内容
- 保持快速（最小化交互）

### 2. kickoff - 项目启动工作流

**触发方式**: `/kickoff [idea/file-path]`

**目标**: 将想法或收件箱笔记转化为结构化的项目笔记。

**工作流步骤**:

**阶段1: 启动规划代理**

1. **解析输入**
   - 文件路径：读取收件箱笔记内容
   - 内联文本：直接使用描述
   - 无输入：列出 `00_收件箱/` 文件供用户选择

2. **生成规划代理（使用 Task 工具）**
   ```
   subagent_type: "general-purpose"
   description: "Plan project kickoff"
   ```

3. **规划代理任务**:
   - 收集上下文：搜索 `20_项目/` 和 `10_日记/` 中相关笔记
   - 识别相关领域（SoftwareEngineering, Finance, Health 等）
   - 创建计划文件 `90_计划/Plan_YYYY-MM-DD_Kickoff_<ProjectName>.md`
   - 计划内容包括：
     - 来源（收件箱文件或内联输入）
     - 目标（一句话总结）
     - 项目结构（领域、类型、规模）
     - 建议行动项
     - 项目大纲草案（背景、行动阶段、成功指标）
     - 澄清问题（时间线、优先级、约束）
   - 返回计划文件路径

4. **通知用户审阅**
   - 中文提示：请查看计划并按需修改，确认后继续执行

**阶段2: 启动执行代理（用户确认后）**

1. **生成执行代理（全新上下文）**
   ```
   subagent_type: "general-purpose"
   description: "Execute project kickoff"
   ```

2. **执行代理任务**:
   - 读取计划文件
   - 注意用户修改或回答的澄清问题
   - 创建项目笔记：
     - 小型项目：`20_项目/<ProjectName>.md`
     - 中大型项目：`20_项目/<ProjectName>/<ProjectName>.md`
   - 使用 C.A.P. 结构：
     - **背景**: 目标、背景、重要性
     - **行动**: 阶段/里程碑及任务
     - **进展**: 空白区域供未来更新
   - 在今日日记 `10_日记/YYYY-MM-DD.md` 中链接项目
   - 归档计划：移动到 `90_计划/归档/`
   - 如来自收件箱：
     - 更新收件箱文件 frontmatter：`status: processed`, `archived: YYYY-MM-DD`
     - 移动到 `99_系统/归档/收件箱/YYYY/MM/`
     - 创建年月目录（如不存在）

3. **报告结果**
   - 中文总结：项目笔记位置、项目结构、收件箱归档路径、建议下一步

**使用的工具**:
- **Task**: 生成规划和执行子代理
- **Read**: 读取收件箱文件、计划文件、现有项目
- **Write**: 创建计划文件、项目笔记
- **Edit**: 更新收件箱文件 frontmatter
- **Bash**: 搜索相关笔记、移动文件、创建目录

**文件操作模式**:
- **读取**: `00_收件箱/*.md`, `20_项目/**/*.md`, `10_日记/*.md`, `90_计划/Plan_*.md`
- **创建**: `90_计划/Plan_YYYY-MM-DD_Kickoff_<ProjectName>.md`, `20_项目/<ProjectName>.md`
- **修改**: `00_收件箱/[SourceFile].md` (frontmatter)
- **移动**: `90_计划/Plan_*.md` → `90_计划/归档/`, `00_收件箱/*.md` → `99_系统/归档/收件箱/YYYY/MM/`

**关键设计原则**:
- **双代理分离**: 规划专注结构，执行专注创建
- **全新上下文**: 执行代理仅读取计划文件，保持上下文清洁
- **用户检查点**: 用户可在执行前修改计划
- **更好的项目**: 规划阶段确保深思熟虑的结构
- **避免重复**: 检查项目是否已存在

### 3. research - 深度研究工作流

**触发方式**: `/research <topic>`

**目标**: 对技术、概念或复杂主题进行深度研究，创建结构化的研究笔记和原子化的知识库概念。

**工作流步骤**:

**阶段1: 启动规划代理**

1. **解析输入**
   - 研究主题（如 "React Server Components", "Consistent Hashing"）
   - 可选：特定问题或目标
   - 可选：相关项目上下文

2. **生成规划代理**
   ```
   subagent_type: "general-purpose"
   description: "Plan research strategy"
   ```

3. **规划代理任务**:
   - 识别上下文：
     - 检查 `20_项目/` 中相关活跃项目
     - 确定相关领域（SoftwareEngineering, Finance, Health 等）
     - 搜索 `30_研究/` 和 `40_知识库/` 避免重复
   - 识别专业角色：扫描 `99_系统/提示词/` 寻找最相关的专业知识
   - 创建计划文件 `90_计划/Plan_YYYY-MM-DD_Research_<Topic>.md`
   - 计划内容包括：
     - 研究目标（用户将理解什么）
     - 发现的上下文（相关领域、现有笔记、相关项目）
     - 研究策略（搜索文档、查找示例、识别关键概念、创建实践示例、查找最佳实践）
     - 输出结构（主笔记路径、原子概念路径、示例/资源路径）
     - 澄清问题（知识水平、项目相关性、理论 vs 示例驱动）
   - 返回计划文件路径

4. **通知用户审阅**
   - 中文提示：请查看研究计划并按需修改，确认后继续执行

**阶段2: 启动执行代理（用户确认后）**

1. **生成执行代理（全新上下文）**
   ```
   subagent_type: "general-purpose"
   description: "Execute research plan"
   ```

2. **执行代理任务**:
   - 读取计划文件，注意用户修改或回答
   - 进行研究：
     - 使用 WebSearch 获取当前信息
     - 使用 WebFetch 阅读文档
     - 收集实践示例
     - 识别可提取的原子概念
   - 创建主研究笔记 `30_研究/<Area>/<Topic>/<Topic>.md`
     - 章节：概述、核心概念、工作原理、示例、最佳实践、常见陷阱、相关阅读、参考资源
   - 创建原子 Wiki 笔记 `40_知识库/<Category>/<ConceptName>.md`
     - 保持简洁（1-3 段）
     - 包含"相关概念"部分
   - 创建可视化地图（如主题复杂）：`<Topic>_Map.canvas`
   - 创建示例（如适用）：保存在 `30_研究/<Area>/<Topic>/examples/`
   - 链接与追踪：
     - 追加到今日日记 `10_日记/YYYY-MM-DD.md`
     - 如与项目相关，在项目的 Progress 部分添加链接
   - 归档：移动计划到 `90_计划/归档/`

3. **报告结果**
   - 中文总结：已创建笔记、核心要点、下一步建议

**使用的工具**:
- **Task**: 生成规划和执行子代理
- **Read**: 读取计划文件、现有研究笔记、Wiki 模板
- **Write**: 创建计划文件、研究笔记、Wiki 笔记、示例文件
- **WebSearch**: 搜索当前信息
- **WebFetch**: 获取文档内容
- **Bash**: 搜索现有笔记、移动文件、创建目录

**文件操作模式**:
- **读取**: `20_项目/**/*.md`, `30_研究/**/*.md`, `40_知识库/**/*.md`, `99_系统/提示词/*.md`, `99_系统/模板/Wiki_Template.md`, `90_计划/Plan_*.md`
- **创建**: `90_计划/Plan_YYYY-MM-DD_Research_<Topic>.md`, `30_研究/<Area>/<Topic>/<Topic>.md`, `40_知识库/<Category>/<Concept>.md`, `30_研究/<Area>/<Topic>/examples/*`
- **修改**: `10_日记/YYYY-MM-DD.md`, `20_项目/<ProjectName>.md` (Progress 部分)
- **移动**: `90_计划/Plan_*.md` → `90_计划/归档/`

**关键设计原则**:
- **全新上下文**: 执行代理专注于研究和写作
- **更好的规划**: 通过先检查现有内容避免重复笔记
- **用户控制**: 用户可在执行前调整策略
- **减少 Token 使用**: 研究在清洁上下文中进行
- **原子化知识**: 提取可复用概念到知识库

### 4. brainstorm - 交互式头脑风暴工作流

**触发方式**: `/brainstorm [topic]`

**目标**: 通过交互式对话帮助用户探索和完善想法，然后选择性地创建项目或整理知识。

**工作流步骤**:

**阶段1: 头脑风暴模式**

1. **角色定位**
   - 提出探索性问题深化理解
   - 建设性地挑战假设
   - 探索多个角度（技术、实践、创意、战略）
   - 通过建议变体和扩展来构建想法
   - 识别与现有 vault 知识的连接
   - 在对话流动中心理追踪洞察

2. **头脑风暴技巧**
   - **5 个为什么**: 深入挖掘动机和根本原因
   - **假设**: 探索替代场景和可能性
   - **魔鬼代言人**: 挑战想法以强化它们
   - **类比**: 与类似概念或问题建立联系
   - **约束**: 询问"如果我们有无限资源？"或"如果我们只有1周？"

3. **对话流程**
   - 从上下文开始：理解用户起点
     - "是什么激发了这个想法？"
     - "你试图解决什么问题？"
     - "这是为谁准备的？"
   - 深入探索：基于回应提出后续问题
   - 捕获洞察：心理记录关键概念、可行想法、开放问题、潜在挑战、相关知识领域
   - 检查 vault 上下文（按需）：
     - 快速搜索 `20_项目/`, `30_研究/`, `40_知识库/`
     - 如相关，用 `[[NoteName]]` 引用现有笔记
     - 建议与用户现有工作的连接

4. **语气**
   - 好奇且充满活力
   - 支持但具有挑战性
   - 创造性和开放思维
   - 关注可能性，而非限制

**阶段2: 综合**

当用户表示准备结束（或自然结论后），提供**头脑风暴总结**（中文）：
- 核心想法（一段话总结）
- 关键洞察（编号列表）
- 可能方向（带简要描述）
- 待解决问题
- 与现有知识的关联

**阶段3: 行动阶段**

提供三个选项（中文）：

1. **创建项目**
   - 使用 Task 工具调用 `/kickoff` 工作流
   - 传递头脑风暴总结作为项目想法
   - 让 kickoff skill 处理项目创建

2. **整理知识**
   - 识别结构：确定相关领域、识别 Wiki 的原子概念、决定主笔记主题
   - 创建笔记：
     - 主参考笔记：`30_研究/<Area>/<Topic>/<Topic>.md`
     - 原子概念：`40_知识库/<Category>/<Concept>.md`
     - 使用头脑风暴洞察填充内容
   - 链接一切：在相关概念间添加 wikilinks、更新今日日记
   - 中文报告：创建的路径和总结

3. **继续探索**
   - 继续头脑风暴对话
   - 或创建简单的收件箱笔记：`00_收件箱/Brainstorm_YYYY-MM-DD_<Topic>.md`
   - 用户可稍后用 `/kickoff` 或 `/parse-knowledge` 处理

**使用的工具**:
- **Read**: 搜索现有项目、研究、知识库笔记
- **Write**: 创建收件箱笔记、研究笔记、Wiki 笔记
- **Task**: 调用 `/kickoff` 工作流（如用户选择创建项目）
- **Bash**: 搜索 vault 中的相关内容

**文件操作模式**:
- **读取**: `20_项目/**/*.md`, `30_研究/**/*.md`, `40_知识库/**/*.md`, `99_系统/模板/Wiki_Template.md`
- **创建**: `00_收件箱/Brainstorm_*.md`, `30_研究/<Area>/<Topic>/<Topic>.md`, `40_知识库/<Category>/<Concept>.md`
- **修改**: `10_日记/YYYY-MM-DD.md`

**关键设计原则**:
- **保持对话模式**: 头脑风暴期间不要跳到创建文件
- **不要过度工程化**: 这是探索，不是执行
- **心理追踪想法**: 还不要创建 TODO 列表或计划
- **有帮助时引用 vault**: 但不要打断流程
- **原子化 Wiki 笔记**: 每个笔记一个概念

## 技能协同机制

### 双代理模式（kickoff & research）

**架构**:
```
用户 → Orchestrator (你) → Planning Agent → 创建计划文件
                          ↓
                    用户审阅/修改计划
                          ↓
     Orchestrator → Execution Agent → 读取计划 → 执行任务
```

**优势**:
1. **全新上下文**: 执行代理从干净的状态开始，仅读取计划
2. **专注工作**: 规划代理专注结构，执行代理专注创建
3. **用户检查点**: 用户可在项目创建前修改计划
4. **更好的输出**: 规划阶段确保深思熟虑的结构
5. **减少 Token 使用**: 避免在执行阶段携带大量上下文

**实现细节**:
- 使用 Task 工具生成子代理
- 规划代理返回计划文件路径
- Orchestrator 等待用户确认
- 执行代理仅接收计划文件路径作为输入
- 计划文件作为两个代理之间的"契约"

### 技能链式调用（brainstorm → kickoff）

**场景**: 用户在头脑风暴后选择创建项目

**流程**:
```
/brainstorm → 交互式对话 → 综合总结 → 用户选择"创建项目"
                                      ↓
                              生成 kickoff 子代理
                                      ↓
                              传递头脑风暴总结
                                      ↓
                              kickoff 工作流执行
```

**优势**:
- 无缝过渡：从探索到执行
- 保留上下文：头脑风暴洞察传递给 kickoff
- 模块化：每个 skill 保持独立，通过标准接口协作

### 文件作为状态管理

**Frontmatter 驱动的状态**:
- `status: pending|active|on-hold|done|processed`
- `created: YYYY-MM-DD`
- `archived: YYYY-MM-DD`
- `area: "[[AreaName]]"`
- `type: project|reference|plan`

**状态转换示例**:
```
收件箱笔记 (status: pending)
    ↓ /kickoff
项目笔记 (status: active)
    ↓ 工作进行中
项目笔记 (status: done)
    ↓ /archive
归档 (archived: YYYY-MM-DD)
```

**搜索模式**:
- 查找活跃项目：`grep "status: active" 20_项目/**/*.md`
- 查找待处理收件箱：`grep "status: pending" 00_收件箱/*.md`
- 查找停滞项目：检查 `10_日记/` 中最后提及日期

## 工作流模式总结

### 模式1: 单代理直接执行（start-my-day）

**适用场景**: 简单、快速的工作流，不需要用户中间审阅

**特点**:
- 一次性执行
- 最小化用户交互
- 直接文件操作
- 快速反馈

**实现**:
```
用户命令 → 收集上下文 → 用户输入 → 创建/更新文件 → 报告结果
```

### 模式2: 双代理规划-执行（kickoff, research）

**适用场景**: 复杂工作流，需要深思熟虑的规划和用户审阅

**特点**:
- 两阶段执行
- 用户检查点
- 全新执行上下文
- 更好的输出质量

**实现**:
```
用户命令 → Planning Agent → 计划文件 → 用户审阅 → Execution Agent → 报告结果
```

### 模式3: 交互式对话（brainstorm）

**适用场景**: 探索性工作，需要多轮对话和思维发散

**特点**:
- 多轮交互
- 心理追踪洞察
- 延迟文件创建
- 可选的后续行动

**实现**:
```
用户命令 → 对话循环 → 综合总结 → 用户选择行动 → 执行选定工作流
```

### 模式4: 技能组合（brainstorm + kickoff）

**适用场景**: 需要多个 skills 协同完成复杂任务

**特点**:
- 技能链式调用
- 上下文传递
- 模块化设计
- 灵活组合

**实现**:
```
Skill A → 输出/总结 → Skill B (通过 Task 工具) → 最终结果
```

## 最佳实践

### 文件操作

1. **始终检查文件是否存在**: 避免覆盖或重复
2. **使用 frontmatter 管理状态**: 标准化的元数据便于搜索和过滤
3. **保持文件结构一致**: 遵循 OrbitOS 的目录约定
4. **使用 wikilinks 连接笔记**: `[[NoteName]]` 建立知识网络
5. **归档而非删除**: 移动到 `99_系统/归档/` 保留历史

### 工作流设计

1. **单一职责**: 每个 skill 专注一个明确的任务
2. **用户控制**: 在关键决策点提供检查点
3. **全新上下文**: 复杂工作流使用子代理保持上下文清洁
4. **快速反馈**: 最小化等待时间，提供进度更新
5. **错误处理**: 优雅处理边缘情况（文件不存在、空项目列表等）

### 用户体验

1. **使用用户语言**: 所有输出使用中文（根据 CLAUDE.md 规则）
2. **提供上下文**: 解释为什么建议某个行动
3. **可操作的建议**: 提供具体的下一步，而非模糊的指导
4. **保持简洁**: 避免冗长的输出，聚焦关键信息
5. **链接相关资源**: 使用 wikilinks 连接到相关笔记和项目

### 代码质量

1. **使用正确的工具**: Read 用于读取，Write 用于创建，Edit 用于修改
2. **批量操作**: 并行执行独立的文件读取
3. **验证输入**: 检查用户输入的有效性
4. **清晰的错误消息**: 当出错时提供有用的反馈
5. **遵循 Obsidian 格式**: frontmatter 在第1行，使用数组语法，无重复键
