kickoff 双代理工作流示意图
================================

触发: /kickoff [idea/file-path]

═══════════════════════════════════════════════════════════════
阶段1: 规划代理（Planning Agent）
═══════════════════════════════════════════════════════════════

步骤1: Orchestrator 解析输入
┌─────────────────────────────────────────┐
│ 输入类型:                               │
│ ├─ 文件路径: Read 00_收件箱/MyIdea.md  │
│ ├─ 内联文本: "构建习惯追踪应用"         │
│ └─ 无输入: 列出收件箱文件供用户选择    │
└─────────────────────────────────────────┘
           ↓

步骤2: 生成规划代理
┌─────────────────────────────────────────┐
│ Task 工具:                              │
│   subagent_type: "general-purpose"      │
│   description: "Plan project kickoff"   │
│   prompt: "创建项目启动计划..."         │
└─────────────────────────────────────────┘
           ↓

步骤3: 规划代理执行
┌─────────────────────────────────────────┐
│ 1. 收集上下文:                          │
│    - Grep: 20_项目/**/*.md (相关项目)   │
│    - Grep: 10_日记/**/*.md (相关提及)   │
│                                         │
│ 2. 识别领域:                            │
│    - 扫描: 30_研究/ (相关领域)          │
│    - 确定: SoftwareEngineering, Health等│
│                                         │
│ 3. 创建计划文件:                        │
│    Write: 90_计划/Plan_2026-03-05_      │
│           Kickoff_习惯追踪应用.md       │
│                                         │
│    内容结构:                            │
│    - 来源: 00_收件箱/MyIdea.md          │
│    - 目标: 构建个人习惯追踪应用         │
│    - 项目结构:                          │
│      * 领域: [[SoftwareEngineering]]    │
│      * 类型: project                    │
│      * 规模: 中型（文件夹）             │
│    - 建议行动项: [定义成功标准, ...]    │
│    - 项目大纲草案:                      │
│      * 背景: [问题、重要性]             │
│      * 行动: [阶段1, 阶段2, ...]        │
│      * 成功指标: [指标1, 指标2, ...]    │
│    - 澄清问题:                          │
│      * 时间线？                         │
│      * 优先级？                         │
│      * 约束？                           │
│                                         │
│ 4. 返回路径:                            │
│    return: "90_计划/Plan_2026-03-05_    │
│             Kickoff_习惯追踪应用.md"    │
└─────────────────────────────────────────┘
           ↓

步骤4: Orchestrator 通知用户
┌─────────────────────────────────────────┐
│ "我已在 `90_计划/Plan_2026-03-05_       │
│  Kickoff_习惯追踪应用.md` 创建了项目    │
│  启动计划。请查看并按需修改，确认后     │
│  继续执行。"                            │
└─────────────────────────────────────────┘
           ↓
    [用户审阅计划]
           ↓
    [用户确认: "继续"]
           ↓

═══════════════════════════════════════════════════════════════
阶段2: 执行代理（Execution Agent）- 全新上下文
═══════════════════════════════════════════════════════════════

步骤1: Orchestrator 生成执行代理
┌─────────────────────────────────────────┐
│ Task 工具:                              │
│   subagent_type: "general-purpose"      │
│   description: "Execute project kickoff"│
│   prompt: "执行位于以下路径的项目启动   │
│            计划: 90_计划/Plan_2026-03-05│
│            _Kickoff_习惯追踪应用.md"    │
│                                         │
│ 注意: 仅传递计划文件路径，无其他上下文  │
└─────────────────────────────────────────┘
           ↓

步骤2: 执行代理执行
┌─────────────────────────────────────────┐
│ 1. 读取计划:                            │
│    Read: 90_计划/Plan_2026-03-05_       │
│          Kickoff_习惯追踪应用.md        │
│    - 注意用户修改                       │
│    - 注意回答的澄清问题                 │
│                                         │
│ 2. 创建项目笔记:                        │
│    根据规模决定路径:                    │
│    ├─ 小型: 20_项目/习惯追踪应用.md     │
│    └─ 中大型: 20_项目/习惯追踪应用/     │
│               习惯追踪应用.md           │
│                                         │
│    Write: 20_项目/习惯追踪应用/         │
│           习惯追踪应用.md               │
│                                         │
│    Frontmatter:                         │
│      title: "习惯追踪应用"              │
│      type: project                      │
│      created: 2026-03-05                │
│      status: active                     │
│      area: "[[SoftwareEngineering]]"    │
│      due: 2026-04-01                    │
│      priority: P1                       │
│      tags: [project, app, habits]       │
│                                         │
│    内容结构（C.A.P.布局）:              │
│    # 习惯追踪应用                       │
│                                         │
│    ## Context (背景)                    │
│    [目标、背景、重要性]                 │
│                                         │
│    ## Actions (行动)                    │
│    ### 阶段1: 设计                      │
│    - [ ] 定义核心功能                   │
│    - [ ] 设计数据模型                   │
│                                         │
│    ### 阶段2: 开发                      │
│    - [ ] 搭建项目框架                   │
│    - [ ] 实现核心功能                   │
│                                         │
│    ## Progress (进展)                   │
│    [留空供未来更新]                     │
│                                         │
│ 3. 链接到今日日记:                      │
│    Edit: 10_日记/2026-03-05.md          │
│    追加: "- 启动新项目 [[习惯追踪应用]]"│
│                                         │
│ 4. 归档计划:                            │
│    Bash: mv 90_计划/Plan_2026-03-05_    │
│             Kickoff_习惯追踪应用.md     │
│             90_计划/归档/               │
│                                         │
│ 5. 处理收件箱（如适用）:                │
│    如来自: 00_收件箱/MyIdea.md          │
│    a. 更新 frontmatter:                 │
│       Edit: 00_收件箱/MyIdea.md         │
│       status: pending → processed       │
│       archived: 2026-03-05              │
│    b. 移动文件:                         │
│       Bash: mkdir -p 99_系统/归档/      │
│                      收件箱/2026/03/    │
│             mv 00_收件箱/MyIdea.md      │
│                99_系统/归档/收件箱/     │
│                2026/03/                 │
└─────────────────────────────────────────┘
           ↓

步骤3: 执行代理报告结果

│ ## 项目创建完成                         │
│                                         │
│ **项目笔记:** [[习惯追踪应用]] 位于    │
│ 20_项目/习惯追踪应用/                   │
│                                         │
│ **项目结构:** 中型项目，包含独立文件夹  │
│                                         │
│ **收件箱归档:** 99_系统/归档/收件箱/    │

│                                         │

│ - [ ] 研究现有习惯追踪应用的最佳实践    │
│ - [ ] 设计数据模型和API                 │
└─────────────────────────────────────────┘
           ↓

步骤4: Orchestrator 转发结果给用户

═══════════════════════════════════════════════════════════════
文件操作总结
═══════════════════════════════════════════════════════════════

规划代理:
- Read: 00_收件箱/MyIdea.md
- Grep: 20_项目/**/*.md, 10_日记/**/*.md, 30_研究/**/*.md
- Write: 90_计划/Plan_2026-03-05_Kickoff_习惯追踪应用.md

执行代理:
- Read: 90_计划/Plan_2026-03-05_Kickoff_习惯追踪应用.md
- Write: 20_项目/习惯追踪应用/习惯追踪应用.md
- Edit: 10_日记/2026-03-05.md, 00_收件箱/MyIdea.md
- Bash: mv (归档计划和收件箱), mkdir -p (创建归档目录)

═══════════════════════════════════════════════════════════════
关键特点
═══════════════════════════════════════════════════════════════

✓ 双代理分离: 规划专注结构，执行专注创建
✓ 全新上下文: 执行代理仅读取计划文件
✓ 用户检查点: 用户可在执行前修改计划
✓ 更好的项目: 规划阶段确保深思熟虑的结构
✓ 避免重复: 规划阶段检查现有项目
✓ 完整归档: 计划和收件箱都被妥善归档

═══════════════════════════════════════════════════════════════
双代理模式的优势
═══════════════════════════════════════════════════════════════

1. 上下文隔离: 执行代理不携带规划阶段的大量上下文
2. Token 效率: 减少不必要的上下文传递
3. 清晰职责: 每个代理有明确的单一职责
4. 用户控制: 在关键决策点提供审阅机会
5. 更好质量: 规划阶段确保结构合理性
