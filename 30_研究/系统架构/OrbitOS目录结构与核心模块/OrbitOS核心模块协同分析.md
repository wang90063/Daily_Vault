# OrbitOS 核心模块协同分析

## 概述

OrbitOS 是一个基于 Obsidian 的知识管理与任务执行系统，通过数字前缀的规范化目录结构（00-99）实现从"信息捕捉"到"项目执行"再到"知识沉淀"的完整闭环。系统的核心理念是"一切围绕用户运转"（Everything orbits around you），通过 Claude Code CLI 的 Agent Skills 驱动自动化工作流，将非结构化的想法转化为结构化的知识资产。

本文档深入拆解每个核心模块的功能逻辑、流转机制，以及模块间的协同关系，并提供具体的文件样例以辅助理解。

---

## 核心模块拆解

### 00_收件箱：快速捕捉与智能分流

**功能定位**
收件箱是系统的第一入口，用于快速捕捉所有未分类的想法、任务或信息碎片。它遵循 GTD (Getting Things Done) 方法论中的"捕捉"原则，确保用户不会因为思考"这个想法该放哪里"而打断创作流。

**处理策略**
1. **捕捉阶段**: 使用 `Inbox_Template.md` 快速创建条目，标记 `status: captured`
2. **处理阶段**: 通过 `/kickoff` (转化为项目) 或 `/research` (转化为研究) 进行分流
3. **归档阶段**: 处理完成后标记 `status: processed`，定期通过 `/archive` 移动到 `99_系统/归档/收件箱/YYYY/MM/`

**归档逻辑**
- 按时间归档：`99_系统/归档/收件箱/2026/03/`
- 保留 frontmatter 元数据，便于后续检索
- 归档不等于删除，而是"冷存储"，保持系统整洁的同时保留历史轨迹

**文件样例**

```markdown
# 文件: 00_收件箱/探索项目的技术细节.md
---
type: inbox
created: 2026-03-03
topic: 技术探索
status: captured
source: 个人思考
tags: [inbox, tech]
---

1. 项目来源：这个项目整体是从 https://github.com/MarsWang42/OrbitOS clone 下来的
2. 我想知道这个项目是怎么跟 Claude Code CLI 进行交互，驱动 Claude 去完成项目目的的
3. 我想，首先要搞清楚这个项目的结构，都分哪些模块
4. 每个模块都是用来干什么的
5. 最后，我想知道我该怎么很好滴使用这个项目，来完成我每日的工作
```

**处理后流转**
- 通过 `/kickoff` → 转化为 `20_项目/项目结构与原理深度探索.md`
- 通过 `/research` → 生成 `90_计划/Plan_2026-03-04_Research_XXX.md`
- 标记 `status: processed` → 归档到 `99_系统/归档/收件箱/2026/03/`

### 10_日记：时间轴与项目追踪中枢

**功能定位**
日记是系统的时间锚点，记录每日的计划、执行和反思。它不仅是个人日志，更是项目进展的追踪器和上下文切换的缓冲区。

**日志记录机制**
- 文件命名：`YYYY-MM-DD.md`（如 `2026-03-04.md`）
- 使用 `/start-my-day` 技能自动生成当日日记
- 通过 wikilinks 链接到活跃项目，形成双向追踪

**项目追踪逻辑**
- 日记中的"相关项目"部分列出当日涉及的项目
- 项目文件的"进展"部分反向链接到日记
- 形成时间线视图：从日记可以看到"今天做了什么"，从项目可以看到"这个项目的历史演进"

**文件样例**

```markdown
# 文件: 10_日记/2026-03-04.md
---
date: 2026-03-04
day: Wednesday
week: 10
---
# 2026-03-04

## 待办事项

- [x] 【项目推进】推进活跃项目: [[项目结构与原理深度探索]] (Task 1.2 已完成，详见 [[OrbitOS交互机制分析]])。
```

**与项目的协同**
- 日记中的待办事项直接链接到项目：`[[项目结构与原理深度探索]]`
- 项目的 Progress 部分记录：`- 2026-03-04: [[2026-03-04]] - 完成 Task 1.2`
- 通过 `/start-my-day` 自动读取昨日日记和活跃项目，生成今日建议

### 20_项目：C.A.P. 结构驱动的执行引擎

**功能定位**
项目目录是系统的执行核心，存放所有活跃的攻坚点。采用扁平化管理（不按领域分文件夹），通过 frontmatter 的 `area` 字段链接到知识库领域。

**C.A.P. 布局详解**
- **Context (背景)**: 定义项目目标、成功指标、限制条件
- **Actions (行动)**: 分阶段的可执行任务清单
- **Progress (进展)**: 时间线记录，链接到每日日记

**组织逻辑**
- 简单项目：单个 `.md` 文件
- 复杂项目：创建文件夹（当资产文件 ≥ 5 个时）
- 通过 `area: "[[SE_CodeBase]]"` 链接到知识库领域，而非文件夹层级
